# ScamGuard Backend

Express + Postgres backend for the ScamGuard browser extension: cloud scam checks
(urlscan.io, Google Safe Browsing, HTTPS probe, domain heuristics), crowdsourced
community reports, anonymous UUID identity, daily quotas, and contributor credits.

## Setup

```sh
npm install
cp .env.example .env   # fill in DATABASE_URL and API keys
npm run migrate        # applies schema.sql
npm start
```

On Railway: add a Postgres plugin (provides `DATABASE_URL`), set
`URLSCAN_API_KEY` and `GOOGLE_SAFE_BROWSING_API_KEY` in service variables,
and run `npm run migrate` once via `railway run`.

To import data from the old SQLite database:

```sh
npm install --no-save better-sqlite3
node scripts/import-sqlite.js path/to/old.db
```

## Identity

Every `/api/*` request must send an `X-User-UUID` header containing a UUID v4
generated by the extension (`crypto.randomUUID()`). First sighting auto-creates
the user. This is anonymous identity, not authentication.

## Quotas & tiers

- Free users: 50 cloud checks per UTC day. Cache hits are free.
- Premium (paid `tier` or earned premium days): unlimited.
- Earned premium days: 1 day is consumed on the first request of each UTC day
  while the user has days remaining and no paid subscription.
- Contributor credits (anti-abuse rules):
  - Only `scam` reports can earn credits (rewarding `safe` agreement would be
    trivially farmable on popular legit domains).
  - Corroboration counts only from accounts 7+ days old, needs the reporter
    plus 3+ others, and the agreeing reports must span 24+ hours.
  - First threshold crossing triggers validation against Safe Browsing and
    urlscan. Externally confirmed -> credits auto-paid. Otherwise the domain
    enters a review queue (`node scripts/review-domain.js list|confirm|reject`)
    and credits are paid only on manual confirmation.
  - Max 1 day per domain per user, and max 4 earned days per rolling 30 days.

## Community verdicts

Community reports alone can never produce a `scam` verdict. External sources
(Safe Browsing, urlscan) can assert `scam` on their own. A community scam
consensus — 5+ tenured reporters, 80%+ agreement, spread over 24+ hours —
yields at most `suspicious` (with `community_flagged: true` in the check
response) until the domain is confirmed via external validation or manual
review. Domains rejected in review have their community signal ignored.

## API

All endpoints require `X-User-UUID`. Verdicts: `scam | safe | suspicious`.
Domains are normalized server-side to the registrable domain (eTLD+1, via
the Public Suffix List) — `login.evil.co.uk` and `pay.evil.co.uk` are the
same domain everywhere, so subdomain rotation can't split community reports.

| Endpoint | Description |
|---|---|
| `GET /api/v1/check?domain=` | Cloud check (cached, cap-enforced). `local_fallback: true` + HTTP 429 when over cap. |
| `GET /api/v1/reports?domain=` | Recent reports (domain, verdict, timestamp; comments are not public), optionally filtered. |
| `POST /api/v1/reports` | `{domain, verdict, comment?}` — one report per user per domain (re-reporting overwrites). |
| `GET /api/v1/user/status` | Tier, effective tier, premium days, checks used/remaining. |
| `DELETE /api/v1/user` | Erase all data tied to this UUID (reports, credits, user row). |
| `GET /health` | Health check, no UUID needed. |

## Privacy & abuse protections

- Per-IP rate limits (300 req/15min on `/api`, 20 report POSTs/hour) backstop
  the per-UUID quota, which alone is bypassable by minting UUIDs.
- No per-user check history is stored anywhere; `domain_cache` is shared and
  has no user column. Never add logging that pairs the UUID header with the
  domain being checked — that would be a browsing-history database.
- No server-side fetches of user-supplied domains (SSRF). The HTTPS signal
  is the extension's job; it already knows how the page loaded.
- `DELETE /api/user` supports right-to-erasure requests.

`GET /api/check` response:

```json
{
  "local_fallback": false,
  "cached": true,
  "checks_remaining": 37,
  "result": { "domain": "...", "verdict": "safe", "checks": { "...": "..." } }
}
```

Page-content heuristics run in the extension (it has the live DOM); the server
handles everything requiring API keys or shared state.
